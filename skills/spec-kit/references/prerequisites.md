# Pre-Command Prerequisites

Check these conditions before recommending each command. If prerequisites are missing, tell the user what to fix first.

## Before `/speckit.constitution`

- [ ] `.specify/` directory exists (run `specify init --here --ai claude` first)
- [ ] No existing `constitution.md` (running again overwrites it — warn the user)

## Before `/speckit.specify <description>`

- [ ] `.specify/memory/constitution.md` exists
- [ ] Description focuses on WHAT/WHY — reject if it contains tech stack terms (framework names, database names, language names)
- [ ] Description is specific enough to generate acceptance scenarios (reject vague terms like "manage", "handle", "improve")

**Flag these patterns as too vague:**

- "improve performance" → ask: which operation? what's the current baseline? what's the goal?
- "manage users" → ask: create? edit? delete? what role restrictions?
- "add notifications" → ask: what triggers them? email/push/in-app? who receives them?

## Before `/speckit.clarify`

- [ ] `spec.md` exists for the current feature
- [ ] Spec has at least one user story written
- Note: capped at **5 questions max** per session — prioritize the most ambiguous areas

## Before `/speckit.plan <tech stack>`

- [ ] `spec.md` exists and has acceptance scenarios (not just an overview)
- [ ] `/speckit.clarify` has been run OR user explicitly confirms spec is clear enough
- [ ] User has provided tech stack in the prompt (language, framework, database, testing tool)

**Prompt for missing tech context:**

- "What language/runtime?" (Node.js, Python, Go, etc.)
- "What framework?" (Next.js, FastAPI, Rails, etc.)
- "What database/storage?" (PostgreSQL, SQLite, S3, none)
- "What testing framework?" (Jest, pytest, etc.)
- "Any constraints?" (no new dependencies, must work offline, existing codebase, etc.)

## Before `/speckit.tasks`

- [ ] `plan.md` exists (hard requirement — command will fail without it)
- [ ] `spec.md` exists
- Recommended: `research.md`, `data-model.md` exist (generated by `/speckit.plan`)
- Recommended: `/speckit.analyze` has been run if plan and spec were written at different times

## Before `/speckit.analyze`

- [ ] `spec.md` exists
- [ ] `plan.md` exists
- [ ] Run after `/speckit.tasks` if `tasks.md` exists (checks cross-artifact consistency)
- Note: **read-only** — safe to run at any time, never modifies files

## Before `/speckit.implement`

- [ ] `tasks.md` exists with at least one `[ ]` task
- [ ] All checklists in `.specify/specs/<feature>/checklists/` are complete (command will pause if not)
- [ ] **Local CLI tools are installed** — the agent will execute tool commands (npm, dotnet, pip, etc.) that appear in `tasks.md`; ensure the correct versions are available
- Recommended: run `/speckit.analyze` first to catch consistency issues
- Note: the command automatically creates/verifies ignore files (`.gitignore`, `.dockerignore`, `.eslintignore`, etc.) based on the detected tech stack before starting implementation
- After implementation: test the running application — runtime errors (e.g., browser console errors) may not appear in CLI output; paste any errors back to the agent for resolution

## Before `/speckit.checklist`

- [ ] `spec.md` exists (the checklist validates requirement quality)
- Note: generates files in `checklists/` subdirectory — run before `/speckit.implement`

## Before `/speckit.taskstoissues`

- [ ] `tasks.md` exists
- [ ] GitHub MCP server is configured (Claude uses MCP tools to create issues — not `gh` CLI)
- [ ] A GitHub remote pointing to GitHub is configured (`git remote -v`)
